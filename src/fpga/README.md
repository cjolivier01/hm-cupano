# Zybo Z7 FPGA Stitching Path

This directory is a first-pass ARM+FPGA replacement for the CUDA kernels that dominate `src/pano/cudaPano.inl`.

## Why Verilog only

Vivado/IP Integrator integration is simpler if the custom blocks share one HDL style. I used Verilog for the FPGA side and kept the ARM-facing control layer in C++. Adding VHDL as a second source language would mostly duplicate the same IP blocks.

## Mapping from the CUDA pipeline

The current soft-seam path in `cudaPano.inl` does six things:

1. Remap image 1 into canvas space.
2. Copy the overlap ROI into the left blend buffer.
3. Remap image 2 into canvas space.
4. Copy the overlap ROI into the right blend buffer.
5. Build Gaussian/Laplacian pyramids and blend them.
6. Paste the blended ROI back onto the canvas.

`panoFpgaPlan.cpp` builds exactly that sequence as ARM-visible stages. `panoFpgaController.cpp` programs those stages into a memory-mapped register bank. The HDL blocks in `hdl/` cover the expensive datapath pieces:

- `pano_remap_core.v`: sequential remap engine with map-driven source fetch requests.
- `pano_downsample2x2.v`: 2x2 Gaussian/downsample operator for two images and the seam mask.
- `pano_laplacian_core.v`: one pyramid level of `high - upsample(low)`.
- `pano_blend_core.v`: mask-weighted Laplacian blend with alpha-aware bypass.
- `pano_reconstruct_core.v`: one pyramid level of `upsample(low) + lap`.
- `pano_kernel_dispatch.v` and `pano_accel_top.v`: ARM-controlled shell that dispatches the selected kernel.
- `pano_axi_single_read.v`, `pano_axi_single_write.v`, `pano_copy_roi_engine.v`, `zybo_pano_remap_engine.v`, and `zybo_pano_accel_top.v`: first-pass Zybo/Zynq transport wrappers around the datapath modules.

## ARM responsibilities

The Zynq PS should:

- capture or decode the two incoming frames,
- allocate physically contiguous DDR buffers for frames, remap tables, and pyramid scratch space,
- provide continuous OpenCV host buffers for the control-mask assets before upload,
- preconvert the seam mask to fixed-point weights,
- bind the logical stitch plan to physical addresses with `BindPlan()`,
- submit each operation through `PanoFpgaController`.

## PL responsibilities

The Zynq PL should:

- stream bulk image data through AXI DMA or a custom AXI master,
- keep full-resolution frames in DDR rather than BRAM,
- feed the per-pixel or per-block operators in `hdl/`,
- raise the corresponding `*_done` signal when the selected stage finishes.

The HDL now includes a first-pass Zybo wrapper around the compute blocks, but it is still a minimal shell rather than a finished Vivado block design. You still need to connect it to the exact PS HP port, DMA, cache-maintenance, and camera ingress topology you plan to ship.

## Fixed-point defaults

- Pyramid datapath samples default to signed `Q9.8` in 18 bits.
- Blend-mask weights default to unsigned `U0.16`.
- Exposure/color adjustments default to signed `Q8.8`.

## Simulation

- `bazelisk test //src/fpga:hdl_iverilog_test` runs the Verilog testbenches through `iverilog` when `iverilog` and `vvp` are installed.
- That test now includes two reduced end-to-end stitch cases:
  - cropped `assets/left.png` + `assets/right.png`
  - cropped `assets/s1.jpg` + `assets/s2.jpg`
- Both cases run the actual Verilog remap/copy/pyramid-blend pipeline in `pano_two_image_assets_pipeline.v` and compare the simulated canvas against a reference image generated by `hdl/tb/gen_two_image_assets_case.py`.
- The reduced test now consumes file-based control assets rather than hard-coded identity maps in RTL. By default the generator writes a reduced control directory under the case temp directory containing:
  - `mapping_0000.tif`, `mapping_0000_x.tif`, `mapping_0000_y.tif`
  - `mapping_0001.tif`, `mapping_0001_x.tif`, `mapping_0001_y.tif`
  - `seam_file.png`
- You can override those with external reduced control directories using:
  - `FPGA_CONTROL_DIR_LEFT_RIGHT=<path>`
  - `FPGA_CONTROL_DIR_S1_S2=<path>`
  - Each path may be repo-relative or absolute.
- Set `FPGA_DUMP_VCD=1` when running `src/fpga/hdl/run_iverilog_tests.sh` or `bazelisk test //src/fpga:hdl_iverilog_test` to emit `left_right.vcd` and `s1_s2.vcd` for GTKWave inspection. With Bazel, the runner copies them into `TEST_UNDECLARED_OUTPUTS_DIR` when that environment variable is available.
- Set `FPGA_ARTIFACT_DIR=<path>` to preserve the per-case generated control files, hex inputs, and simulation outputs outside the temp directory.
- In environments without those tools, the test exits cleanly with a skip message.

## Running Tests

### Full optimized workspace build

Use this first if you want to verify the repo still builds before touching the FPGA path:

```sh
make perf
```

### Default HDL regression

Run the current unit benches plus the two reduced end-to-end stitch cases:

```sh
src/fpga/hdl/run_iverilog_tests.sh
```

Or through Bazel:

```sh
bazelisk test //src/fpga:hdl_iverilog_test
```

### Single case with artifacts

`run_two_image_case.sh` is the easiest manual entrypoint. It generates a self-contained case directory with:

- staged control assets under `control/`
- `left.hex`, `right.hex`, `map*.hex`, `mask_high.hex`
- `expected_canvas.hex`
- `canvas_out.hex`
- `pano_two_image_assets.vcd`

Example:

```sh
src/fpga/hdl/run_two_image_case.sh \
  --left assets/left.png \
  --right assets/right.png \
  --mode iverilog
```

The script prints the artifact directory at the end.

### Parameterized case sizes

The single-case runner can build arbitrary extracted RTL cases instead of the fixed `16x8` default:

```sh
src/fpga/hdl/run_two_image_case.sh \
  --left assets/left.png \
  --right assets/right.png \
  --width 64 \
  --height 32 \
  --overlap 16 \
  --pad 2 \
  --mode iverilog
```

Constraints:

- `height` must be even.
- `overlap + 2 * pad` must be even.
- `0 < overlap < width`.

### Extracting a seam-crossing control window

If you already have a real two-image Hugin/enblend control directory, extract a seam-crossing window that matches the RTL test geometry:

```sh
python3 src/fpga/hdl/tb/gen_two_image_assets_case.py extract-control \
  --left path/to/left.png \
  --right path/to/right.png \
  --control-dir path/to/full_control_dir \
  --outdir /tmp/extracted_control \
  --width 64 \
  --height 32 \
  --overlap 16 \
  --pad 2
```

Then run the case against that extracted control directory:

```sh
src/fpga/hdl/run_two_image_case.sh \
  --left path/to/left.png \
  --right path/to/right.png \
  --control-dir /tmp/extracted_control \
  --mode iverilog
```

### GEM flow

The repo now has an experimental GEM-oriented runner:

```sh
src/fpga/hdl/run_two_image_case.sh \
  --left assets/left.png \
  --right assets/right.png \
  --mode gem \
  --gem-root /tmp/GEM
```

Requirements:

- `cargo`
- `yosys`
- `nvcc`
- a local GEM checkout, for example:

```sh
git clone --depth 1 --branch staged-aig-release https://github.com/NVlabs/GEM.git /tmp/GEM
cd /tmp/GEM
git submodule update --init --recursive
```

On this machine, GEM also needed:

- a current Rust toolchain from `rustup` rather than Ubuntu's older `cargo`
- `UCC_CUDA_GENCODE=75`
- `UCC_CUDA_PTX=75`
- a local patch in `/tmp/GEM/eda-infra-rs/ucc/src/compile.rs` changing `-std=c++14` to `-std=c++17` so CUDA 13's libcu++ would compile

After that, these both worked:

```sh
. "$HOME/.cargo/env"
cd /tmp/GEM
UCC_CUDA_GENCODE=75 UCC_CUDA_PTX=75 cargo run -r --features cuda --bin cut_map_interactive -- --help
UCC_CUDA_GENCODE=75 UCC_CUDA_PTX=75 cargo run -r --features cuda --bin cuda_test -- --help
```

Notes:

- The `gem` mode still runs the `iverilog` driver first, because that driver produces the `input.vcd` GEM consumes.
- The GEM path is currently experimental for the full stitch pipeline. Yosys is flattening the pipeline memories into registers rather than preserving them as mapped RAMs, which makes synthesis substantially heavier than the plain `iverilog` flow.
- The artifact directory keeps `yosys_memory.log`, `yosys_logic.log`, `gem_build.log`, `gem_map.log`, `gem_sim.log`, and `output.vcd` for debugging.
