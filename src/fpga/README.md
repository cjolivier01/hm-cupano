# Zybo Z7 FPGA Stitching Path

This directory is a first-pass ARM+FPGA replacement for the CUDA kernels that dominate `src/pano/cudaPano.inl`.

## Why Verilog only

Vivado/IP Integrator integration is simpler if the custom blocks share one HDL style. I used Verilog for the FPGA side and kept the ARM-facing control layer in C++. Adding VHDL as a second source language would mostly duplicate the same IP blocks.

## Mapping from the CUDA pipeline

The current soft-seam path in `cudaPano.inl` does six things:

1. Remap image 1 into canvas space.
2. Copy the overlap ROI into the left blend buffer.
3. Remap image 2 into canvas space.
4. Copy the overlap ROI into the right blend buffer.
5. Build Gaussian/Laplacian pyramids and blend them.
6. Paste the blended ROI back onto the canvas.

`panoFpgaPlan.cpp` builds exactly that sequence as ARM-visible stages. `panoFpgaController.cpp` programs those stages into a memory-mapped register bank. The HDL blocks in `hdl/` cover the expensive datapath pieces:

- `pano_remap_core.v`: sequential remap engine with map-driven source fetch requests.
- `pano_downsample2x2.v`: 2x2 Gaussian/downsample operator for two images and the seam mask.
- `pano_laplacian_core.v`: one pyramid level of `high - upsample(low)`.
- `pano_blend_core.v`: mask-weighted Laplacian blend with alpha-aware bypass.
- `pano_reconstruct_core.v`: one pyramid level of `upsample(low) + lap`.
- `pano_kernel_dispatch.v` and `pano_accel_top.v`: ARM-controlled shell that dispatches the selected kernel.
- `pano_axi_single_read.v`, `pano_axi_single_write.v`, `pano_copy_roi_engine.v`, `zybo_pano_remap_engine.v`, and `zybo_pano_accel_top.v`: first-pass Zybo/Zynq transport wrappers around the datapath modules.

## ARM responsibilities

The Zynq PS should:

- capture or decode the two incoming frames,
- allocate physically contiguous DDR buffers for frames, remap tables, and pyramid scratch space,
- provide continuous OpenCV host buffers for the control-mask assets before upload,
- preconvert the seam mask to fixed-point weights,
- bind the logical stitch plan to physical addresses with `BindPlan()`,
- submit each operation through `PanoFpgaController`.

## PL responsibilities

The Zynq PL should:

- stream bulk image data through AXI DMA or a custom AXI master,
- keep full-resolution frames in DDR rather than BRAM,
- feed the per-pixel or per-block operators in `hdl/`,
- raise the corresponding `*_done` signal when the selected stage finishes.

The HDL now includes a first-pass Zybo wrapper around the compute blocks, but it is still a minimal shell rather than a finished Vivado block design. You still need to connect it to the exact PS HP port, DMA, cache-maintenance, and camera ingress topology you plan to ship.

## Fixed-point defaults

- Pyramid datapath samples default to signed `Q9.8` in 18 bits.
- Blend-mask weights default to unsigned `U0.16`.
- Exposure/color adjustments default to signed `Q8.8`.

## Simulation

- `bazelisk test //src/fpga:hdl_iverilog_test` runs the Verilog testbenches through `iverilog` when `iverilog` and `vvp` are installed.
- That test now includes a reduced end-to-end stitch case that crops `assets/left.png` and `assets/right.png`, runs the actual Verilog remap/copy/pyramid-blend pipeline in `pano_two_image_assets_pipeline.v`, and compares the simulated canvas against a reference image generated by `hdl/tb/gen_two_image_assets_case.py`.
- In environments without those tools, the test exits cleanly with a skip message.
